### ISS-001：【高】死代码/逻辑矛盾 - `UpstreamEmptyOutputDetail` 的 unreachable 分支

**文件位置**：[upstream_empty.go](file:///Users/wang/Desktop/project/ds2api/ds2api-dev/internal/httpapi/openai/shared/upstream_empty.go#L5-L14)
**所在函数**：`UpstreamEmptyOutputDetail`
**调用链节点**：`finishResponsesNonStreamResult` / `responsesStreamRuntime.finalize` -> `UpstreamEmptyOutputDetail`

**问题描述**：

`UpstreamEmptyOutputDetail` 存在一个永远不会执行的条件分支（`thinking != ""`），该分支输出错误消息 "Upstream account hit a rate limit and returned reasoning without visible output."，暗示"仅有 thinking 输出"是错误状态。

但实际上，该函数的**唯一调用方**在调用前都已经通过 `strings.TrimSpace(thinking) == ""` 条件过滤，因此 `thinking != ""` 的分支永远不会执行。具体来说：

1. [empty_retry_runtime.go:178](file:///Users/wang/Desktop/project/ds2api/ds2api-dev/internal/httpapi/openai/responses/empty_retry_runtime.go#L178) 的在 `finalize` 中：`strings.TrimSpace(finalText) == "" && strings.TrimSpace(finalThinking) == ""` 条件成立后才调用。
2. [upstream_empty.go:17](file:///Users/wang/Desktop/project/ds2api/ds2api-dev/internal/httpapi/openai/shared/upstream_empty.go#L17) 的 `WriteUpstreamEmptyOutputError`：通过 `ShouldWriteUpstreamEmptyOutputError` 守卫，该函数也要求在 text 和 thinking 均为空时才返回 true。

**问题代码片段**：

```go
func UpstreamEmptyOutputDetail(contentFilter bool, text, thinking string) (int, string, string) {
    _ = text
    if contentFilter {
        return http.StatusBadRequest, "Upstream content filtered the response and returned no output.", "content_filter"
    }
    if thinking != "" {  // 永远不会为 true
        return http.StatusTooManyRequests, "Upstream account hit a rate limit and returned reasoning without visible output.", "upstream_empty_output"
    }
    return http.StatusTooManyRequests, "Upstream account hit a rate limit and returned empty output.", "upstream_empty_output"
}
```

**修复建议**：

移除 `thinking != ""` 的 dead branch，简化函数为：

```go
func UpstreamEmptyOutputDetail(contentFilter bool, text, thinking string) (int, string, string) {
    _ = text
    _ = thinking
    if contentFilter {
        return http.StatusBadRequest, "Upstream content filtered the response and returned no output.", "content_filter"
    }
    return http.StatusTooManyRequests, "Upstream account hit a rate limit and returned empty output.", "upstream_empty_output"
}
```

### ISS-002：【中】逻辑 - `fastSkipSSEData` 首次匹配可能提取错误的 path

**文件位置**：[parser.go:36-48](file:///Users/wang/Desktop/project/ds2api/ds2api-dev/internal/sse/parser.go#L36-L48)
**所在函数**：`fastSkipSSEData`
**调用链节点**：`ParseDeepSeekSSELine` -> `fastSkipSSEData` -> `shouldSkipPath`

**问题描述**：

`fastSkipSSEData` 使用 `strings.Index(data, "\"p\":\"")` 查找**第一个**出现的 `"p":"`，并以紧随其后的下一个 `"` 作为 path 结束位置。此实现依赖两个假设：

1. SSE data 中 `"p"` 键始终排在 JSON 对象的最前面。
2. 不存在嵌套的 `{"p": ...}` 结构（如在 `"v"` 值中）。

当前 DeepSeek SSE 格式满足这两个假设，但如果 DeepSeek 将来变更 JSON 键顺序，或在非标准字段（如 `"message"` 嵌套体）中包含 `"p"` 键，则 `fastSkipSSEData` 会提取错误的 path 并调用 `shouldSkipPath`，存在**误跳过有效内容行**的风险。

注意：`shouldSkipPath` 在 `ParseSSEChunkForContentDetailed`（[parser.go:99](file:///Users/wang/Desktop/project/ds2api/ds2api-dev/internal/sse/parser.go#L99)）中也会被调用，因此 `fastSkipSSEData` 返回 false 漏跳过是安全的（仅失去优化）。但返回 true 误跳过则可能丢弃有效内容。

**问题代码片段**：

```go
func fastSkipSSEData(data string) bool {
    idx := strings.Index(data, `"p":"`)       // 首次匹配，可能匹配到嵌套对象
    if idx < 0 {
        return false
    }
    start := idx + 5
    end := strings.Index(data[start:], `"`)   // 可能在嵌套 JSON 中找到错误的结束引号
    if end < 0 {
        return false
    }
    path := data[start : start+end]
    return shouldSkipPath(path)
}
```

**修复建议**：

方案一（推荐）：使用更精确的匹配，确保匹配的是顶层对象的 `"p"` 键：

```go
func fastSkipSSEData(data string) bool {
    const prefix = `{"p":"`
    const altPrefix = `{ "p":"`
    var start int
    if strings.HasPrefix(data, prefix) {
        start = len(prefix)
    } else if strings.HasPrefix(data, altPrefix) {
        start = len(altPrefix)
    } else {
        return false
    }
    end := strings.Index(data[start:], `"`)
    if end < 0 {
        return false
    }
    path := data[start : start+end]
    return shouldSkipPath(path)
}
```

方案二：仅在确定 `data` 中没有嵌套 `{` 时使用 fast path（过于保守，不推荐）。

### ISS-003：【中】健壮性 - 测试模式检测依赖 `os.Args[0]` 文件名匹配

**文件位置**：[stream.go:40-45](file:///Users/wang/Desktop/project/ds2api/ds2api-dev/internal/sse/stream.go#L40-L45)
**所在函数**：`DefaultAccumulateConfig`
**调用链节点**：`StartParsedLinePump` -> `DefaultAccumulateConfig`

**问题描述**：

`DefaultAccumulateConfig()` 通过检查 `os.Args[0]` 是否包含 `.test` 或 `___test` 来决定使用测试配置还是生产配置。这种方式在某些场景下不可靠：

- 使用 `go test -c` 编译出的测试二进制被重命名后，测试检测失效。
- 某些 CI/CD 环境可能修改 testing binary 路径。
- 如果生产二进制碰巧路径中包含 `.test` 字符串，会误触发测试模式（累积功能被禁用）。

**问题代码片段**：

```go
func DefaultAccumulateConfig() AccumulateConfig {
    if strings.HasSuffix(os.Args[0], ".test") || strings.Contains(os.Args[0], "___test") {
        return testAccumulate
    }
    return productionAccumulate
}
```

**修复建议**：

方案一（推荐）：显式传递配置，让调用方决定使用测试还是生产配置。移除全局检测：

```go
func StartParsedLinePump(ctx context.Context, body io.Reader, thinkingEnabled bool, initialType string) (<-chan LineResult, <-chan error) {
    return startParsedLinePumpWithConfig(ctx, body, thinkingEnabled, initialType, productionAccumulate)
}
```

测试中显式传入 `testAccumulate`：

```go
func TestStartParsedLinePumpParsesAndStops(t *testing.T) {
    results, done := startParsedLinePumpWithConfig(context.Background(), body, false, "text", testAccumulate)
    // ...
}
```

方案二：使用 Go 标准库 `testing.Testing()` 或环境变量 `GO_TEST_MODE=1` 作为检测手段，但这些也都有各自的限制。

### ISS-004：【中】性能 - `streamBodyWithContinueState` 每行产生堆分配

**文件位置**：[client_continue.go:157](file:///Users/wang/Desktop/project/ds2api/ds2api-dev/internal/deepseek/client/client_continue.go#L157)
**所在函数**：`streamBodyWithContinueState`
**调用链节点**：`pumpAutoContinue` -> `streamBodyWithContinueState`

**问题描述**：

移除 64KB `bufio.Writer` 后，每行写入改为直接 `pw.Write(append([]byte(text), '\n'))`。`append` 在 `text` 长度超过底层数组容量时会触发堆分配。对于典型的 SSE 行（< 1KB），这通常不是问题，因为 Go 的 `bufio.Scanner` 返回的 `scanner.Text()` 字符串底层可能共享缓冲区。

但在 Cold Start 或 Auto-Continue 高频率触发场景下，这种逐行 `[]byte` 转换产生的分配可能累积。**实际影响较小**，但值得关注。

**问题代码片段**：

```go
if _, err := pw.Write(append([]byte(text), '\n')); err != nil {
    return hadDone, err
}
```

**修复建议**：

保持当前实现（性能影响实测可忽略），但如果后续基准测试显示问题，可使用 `io.WriteString` + 单独的换行写入：

```go
if _, err := io.WriteString(pw, text); err != nil {
    return hadDone, err
}
if _, err := pw.Write([]byte{'\n'}); err != nil {
    return hadDone, err
}
```

或者恢复一个较小缓冲区（如 4KB）的 `bufio.Writer`。

### ISS-005：【低】逻辑 - `fastSkipSSEData` 不支持带空格的 JSON

**文件位置**：[parser.go:37](file:///Users/wang/Desktop/project/ds2api/ds2api-dev/internal/sse/parser.go#L37)
**所在函数**：`fastSkipSSEData`

**问题描述**：

如果 DeepSeek SSE 数据中包含空格（如 `{"p": "response/content", "v": "hi"}`），`strings.Index(data, "\"p\":\"")` 无法匹配，`fastSkipSSEData` 返回 false，行会走 JSON 解析路径。由于 `shouldSkipPath` 在解析后仍会被调用，这不会导致逻辑错误，仅是**优化失效**。

当前 DeepSeek 所有 SSE 输出均为紧凑 JSON（无空格），此问题仅在格式变更时触发。

**修复建议**：

低优先级。如需处理，可使用正则或更宽松的匹配逻辑（如 ISS-002 的修复方案一中也包含了 `{ "p":"` 的匹配）。

### ISS-006：【低】死代码 - 累积模式下可能的空结果发送

**文件位置**：[stream.go:128-148](file:///Users/wang/Desktop/project/ds2api/ds2api-dev/internal/sse/stream.go#L128-L148)
**所在函数**：`flushBuffer`（闭包）
**调用链节点**：`startParsedLinePumpWithConfig` -> `flushBuffer`

**问题描述**：

`flushBuffer` 在 `len(parts) > 0 || toolDetectionThinkingBuffer.Len() > 0` 时发送结果。如果 `parts` 为空（因 thinking buffer 和 text buffer 都被重置为空）且 `toolDetectionThinkingBuffer` 之前已被发送但未重置（实际上它在发送后会被重置），这里存在发送空 `Parts` 结果的可能性。

实际上，由于 `toolDetectionThinkingBuffer` 在每次发送后被重置，且 `parts` 在 `toolDetectionThinkingBuffer.Len() > 0` 时也进入条件，最终发送的 `Parts` 可能为空列表。这不是 bug，因为消费者正确处理空 `Parts`，但可以澄清逻辑。

**修复建议**：

低优先级。如果追求代码清晰度，可以减少 `len(parts) > 0 || toolDetectionThinkingBuffer.Len() > 0` 的冗余：

```go
// 如果 parts 和 detectionParts 都为空，跳过发送
if len(parts) == 0 {
    return  // 注意：此时 toolDetectionThinkingBuffer 也为空
}
```

但当前实现无实际问题，不需要立即修改。

### ISS-007：【低】逻辑 - `[DONE]` 写入未检查 context 取消

**文件位置**：[client_continue.go:122](file:///Users/wang/Desktop/project/ds2api/ds2api-dev/internal/deepseek/client/client_continue.go#L122)
**所在函数**：`pumpAutoContinue`
**调用链节点**：`pumpAutoContinue` -> 最终 `[DONE]` 写入

**问题描述**：

当自动继续循环结束时，代码通过 `io.Copy(pw, bytes.NewBufferString("data: [DONE]\n"))` 发送最终 `[DONE]` 标记。但此时未检查 `ctx.Done()`。如果客户端已断开连接（context 已取消），`io.Copy` 会因写入到关闭的 pipe 而返回错误，这在多数情况下是可接受的，但会导致日志中出现误导性的 "broken pipe" 错误。

**问题代码片段**：

```go
if hadDone {
    if _, err := io.Copy(pw, bytes.NewBufferString("data: [DONE]\n")); err != nil {
        _ = pw.CloseWithError(err)
    }
}
```

**修复建议**：

添加 context 检查：

```go
if hadDone {
    select {
    case <-ctx.Done():
        _ = pw.CloseWithError(ctx.Err())
    default:
        if _, err := io.WriteString(pw, "data: [DONE]\n"); err != nil {
            _ = pw.CloseWithError(err)
        }
    }
}
```
